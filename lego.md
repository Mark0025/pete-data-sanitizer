## lego.md — how this project works (Lego factory explanation)

### The big idea
Think of this project like a **Lego factory**:

- **A run** = one factory shift.
- **Modules/files** = stations in the factory (load, dedupe, match sellers, export, report).
- **RunTracker** = the **clipboard/scoreboard** that records what happened during the shift (what ran, how long, counts, errors).
- **Outputs** = the finished boxes you deliver (XLSX/CSV) plus the QA reports.

### What you do most of the time
You usually run **one command**:

- **Generate the import files + reports** (no server needed):

```bash
uv run python -m pete_dm_clean build
```

This produces the clean DealMachine import file(s) and the reports on disk.

### Optional config.yaml (stable defaults)
If you don’t want to keep typing flags, create a `config.yaml` at the repo root.

- Start with `config.example.yaml` → copy to `config.yaml`.
- The CLI auto-loads `./config.yaml` if it exists.
- Flags always override config values.

### When you start the server (and why)
The server is only for **viewing** (like a dashboard), not for generating files.

- Start the viewer server:

```bash
uv run python -m pete_dm_clean serve
```

- Then open:
  - `http://127.0.0.1:8765/` (diagram index + generate link)
  - `http://127.0.0.1:8765/runs/latest` (latest run summary)

**Why you don’t need it to build:** `build` writes files directly. The server just reads those files and shows them.

### What gets generated (files you care about)
After `build`, you will see:

- **Import-ready XLSX (upload this to DealMachine)**:
  - `uploads/PETE.DM.FERNANDO.CLEAN.<date>.xlsx`
  - Import sheet is `Sheet1`
- **Import CSV (optional)**:
  - `uploads/PETE.DM.FERNANDO.CLEAN.<date>.csv`
- **Seller summary CSV (review / QA)**:
  - `uploads/PETE.DM.FERNANDO.CLEAN.<date>.seller_summary.csv`

### Where logs, runs, and diagrams go (standard folders)
This project stores “run truth” and diagnostics under `uploads/`:

- **Run records (runtime truth)**: `uploads/runs/`
  - `<run_id>.json` (machine-readable)
  - `<run_id>.summary.md` (human-readable)
  - `<run_id>.log` (loguru log file for debugging)

- **Diagrams**: `uploads/flowcharts/`
  - `acki_run_<run_id>.flow.txt` (runtime-based “Acki-style” pipeline diagram)
  - `acki_run_<run_id>.summary.md` (same info as the run summary)
  - `pipeline_build_staging_*.flow.txt` (older / noisier code-structure diagrams; less useful)

### “Acki-style” diagrams (what they are)
There are two kinds of diagrams in the repo:

- **Runtime-based Acki diagram (recommended)**:
  - Generated from the run record (steps + metrics)
  - Shows the pipeline steps and real counts (rows, unique addresses, duplicates removed)

- **Code-structure diagram (not recommended for understanding)**:
  - Generated by parsing Python source
  - Can include noisy lines like `from __future__ import annotations`

If you want real understanding of what happened, focus on:
- `uploads/runs/<run_id>.summary.md`
- `uploads/flowcharts/acki_run_<run_id>.flow.txt`

### The “one tracker per run” rule (why it matters)
You do **not** want every module to create its own tracker.

- **Correct**: create **one** tracker at the entrypoint (the start of the run) and reuse it.
- **Wrong**: every module creates a tracker “just because” → fragmented truth and confusion.

Stations should either:
- receive `tracker` explicitly, or
- use context helpers (`set_tracker/get_tracker`) when passing it everywhere is too noisy.

### Learning mode: “show me every function call”
This is intentionally **off by default** because it’s noisy and slower.

- Turn it on:

```bash
uv run python -m pete_dm_clean build --trace-calls --trace-max-events 200
```

Outputs go to `uploads/runs/`:
- `<run_id>.calls.txt` (sequential call log; capped)
- `<run_id>.call_counts.json` (aggregate call counts)

### When to randomize External Ids
If DealMachine imports “only some” rows, a common cause is duplicate/blank `External Id`.

- Randomize `External Id` for each output row:

```bash
uv run python -m pete_dm_clean build --randomize-external-ids
```

### “When and why to start what?” (quick table)

- **I want fresh files to upload** → run `build`
- **I want to look at diagrams and run summaries in a browser** → start `serve`
- **I want a diagram artifact on disk** → run `diagram` (or just run `build`, which generates the runtime Acki diagram)
- **I want deep learning/tracing** → run `build --trace-calls` (expect noise)
